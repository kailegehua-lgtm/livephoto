# Screen Flicker Capture PRD

## 1. 文档信息

- 产品名称：Screen Flicker Capture
- 版本：V0.2
- 状态：Draft
- 日期：2026-05-20
- 负责人：待定

## 2. 背景

本项目面向工厂环境，用于抓拍二手手机屏幕的瞬时闪屏缺陷。

现有痛点：

1. 闪屏出现随机且持续时间极短。
2. 当人工或检测系统下达抓拍指令时，缺陷经常已经发生。
3. 系统原生 Live Photo 的固定回溯窗口无法满足产线抓拍容错率要求。

当前产品目标不再是泛用“动态照片”，而是一个稳定、可长期运行的产线测试辅助工具，核心诉求是：

1. 预览必须稳定，不能因为回溯逻辑导致明显卡顿。
2. 支持自定义时长的回溯抓拍，默认目标为前 3 秒、后 1 秒。
3. 允许在后台完成动图/视频导出，导出完成后再允许查看动态效果。
4. 输出结果为本地视频文件，便于后续 CV 或人工复判。

## 3. 目标

### 3.1 产品目标

提供一套 App 内可用的闪屏抓拍能力，支持：

1. 实时摄像头预览。
2. 接收拍摄触发后保存一张主图。
3. 在实验模式下，保存触发前 3 秒、后 1 秒的动态视频。
4. 在稳定模式下，先保证主图和触发后 1 秒视频的可靠落盘。
5. 在 App 内浏览历史记录，并在导出完成后查看动态片段。

### 3.2 工程目标

1. 在 `iPhone 15 Pro` 上支持 `720p + 30fps` 运行。
2. 预览主链路不卡顿。
3. 后台导出不阻塞主线程和视频采集队列。
4. 对内存占用有明确上限和降级策略。

## 4. 产品策略

项目分为两种模式并行推进：

### 4.1 稳定模式

目标是先提供可长期运行、可量产验证的闭环：

1. 稳定预览。
2. 主图拍摄。
3. 触发后 1 秒动态视频。
4. 列表可见。
5. 详情可播放。

该模式是当前默认模式，用于兜底和产线可用性验证。

### 4.2 实验模式

目标是验证最终业务诉求：

1. 前 3 秒回溯。
2. 后 1 秒补录。
3. 合成约 4 秒视频。

实验模式允许存在以下现实约束：

1. 预览可能不如稳定模式顺滑。
2. 导出可能有更长的后台处理时间。
3. 失败率可能高于稳定模式。

## 5. 非目标

以下内容不在本阶段范围内：

1. 不做系统标准 Live Photo 导出。
2. 不做系统相册原生兼容。
3. 不做复杂编辑能力，如裁剪、滤镜、封面重选。
4. 不做分享链路适配。
5. 不做云同步。
6. 不做多机协同。

## 6. 核心用户场景

### 6.1 产线场景

1. 工位上放置被测手机。
2. 检测手机持续预览被测屏幕。
3. 当人工或检测逻辑判断疑似闪屏时，下达抓拍触发。
4. App 保存触发前后的视频片段，供后续复判。

### 6.2 典型流程

1. 用户进入拍摄页。
2. 选择 `稳定模式` 或 `实验模式`。
3. App 开始实时预览。
4. 用户点击拍摄，或后续接入外部触发信号。
5. App 立即记录主图，并进入后台处理。
6. 列表出现一条新记录。
7. 导出完成前，记录处于处理中状态。
8. 导出完成后，用户进入详情页查看动态片段。

## 7. 功能范围

### 7.1 拍摄页

必须支持：

1. 实时相机预览。
2. 模式切换：`稳定模式` / `实验模式`。
3. 快门按钮。
4. 拍摄中状态提示。
5. 保存完成提示。
6. 实验模式的风险提示。

本阶段可选但不强制：

1. 前后摄切换。
2. 曝光时长调节 UI。
3. FPS 调节 UI。
4. 手动锁曝光/白平衡/焦点 UI。

### 7.2 列表页

必须支持：

1. 展示历史记录。
2. 每条记录显示封面图、拍摄时间、模式标记。
3. 每条记录显示处理状态。
4. 支持进入详情页。

### 7.3 详情页

必须支持：

1. 默认显示静态主图。
2. 在导出完成后长按播放动态片段。
3. 动态视频缺失时降级为仅显示主图。

### 7.4 后台处理

必须支持：

1. 拍摄后立即返回轻量反馈，不阻塞预览。
2. 在后台完成视频拼接与导出。
3. 导出完成后更新记录状态。

## 8. 处理状态设计

每条记录至少有以下状态：

1. `captured`
   说明：主图已生成，动态视频尚未完成。
2. `processing`
   说明：正在后台导出视频。
3. `ready`
   说明：动态视频已可播放。
4. `failed`
   说明：导出失败，但主图仍保留。

交互要求：

1. 记录应尽快出现在列表中。
2. `ready` 前不强制提供动态播放能力。
3. 导出失败时，列表仍保留记录并可见失败状态。

## 9. 关键技术要求

### 9.1 预览优先

这是项目的最高约束：

1. 预览主链路不能因为回溯缓存、视频导出而卡顿。
2. 稳定模式必须始终保持预览流畅。
3. 实验模式允许轻微影响，但不能出现明显卡死。

### 9.2 默认采集参数

当前默认参数为：

1. 分辨率：`720p`
2. 帧率：`30fps`
3. 回溯：`3秒`
4. 补录：`1秒`
5. 输出：本地 H.264/HEVC 视频文件

### 9.3 内存与性能约束

1. 必须使用 YUV420 视频格式，禁止 BGRA。
2. 不能在主线程进行高频 sample buffer 拷贝、拼接、写盘。
3. 导出必须使用独立串行队列。
4. Ring buffer 需要固定容量和明确淘汰策略。
5. 必须预留降级空间，包括缩短回溯时长、降低采样帧率、切回稳定模式。

## 10. 当前已验证结果

截至本 PRD 版本，已有以下结论：

1. `稳定模式` 已实现可用闭环。
2. 预览已恢复稳定。
3. 列表与详情页播放已打通。
4. 直接把 pre-roll 缓存常驻挂在预览主链路上，会显著增加卡顿风险。

因此后续工程方向明确为：

1. 保持稳定模式不回退。
2. 将实验模式的 pre-roll 采集模块与稳定模式隔离。
3. 后台处理完成后再开放动态查看。

## 11. 参考实现策略

GitHub 上没有明显可直接套用的成熟“闪屏回溯抓拍”成品方案，但有可借鉴模块：

### 11.1 可借鉴模块

1. `NextLevel`
   用于参考相机会话组织、设备控制、输出管理。
2. `HaishinKit`
   用于参考实时采集、缓冲、编码管线与队列模型。
3. Apple AVFoundation 样例与低层 `AVAssetWriter` demo
   用于参考 sample buffer 处理和时间轴重映射。

### 11.2 自研部分

以下能力需要按产线场景自研：

1. 前 3 秒 ring buffer。
2. 触发冻结与后 1 秒补录。
3. 历史帧与后续帧无缝拼接。
4. 面向 OOM 的内存护栏。
5. 稳定模式与实验模式的双轨切换。

## 12. 成功指标

### 12.1 稳定模式

1. 预览连续运行 5 分钟无明显卡顿。
2. 拍摄成功后列表新增记录。
3. 详情页可播放 1 秒动态片段。
4. 导出失败时不会导致 App 崩溃。

### 12.2 实验模式

1. 预览仍可接受。
2. 能输出前 3 秒 + 后 1 秒的视频文件。
3. 记录能在列表中正确标记为实验模式。
4. 导出完成后可回放约 4 秒片段。

### 12.3 观察指标

1. 平均单条视频导出耗时。
2. 峰值内存占用。
3. 连续触发稳定性。
4. 设备温度与耗电。
5. 实验模式相对稳定模式的掉帧和失败率。

## 13. 验收标准

本阶段验收通过的标准为：

1. 稳定模式可在真机上稳定运行。
2. 稳定模式拍摄后能新增记录并播放。
3. 拍摄后允许后台处理，未完成前不要求立即可看动态效果。
4. 实验模式具备独立入口和独立记录标记。
5. 后续可以在不破坏稳定模式的前提下持续迭代 pre-roll。

## 14. 后续方向

下一阶段重点是：

1. 提升实验模式的 pre-roll 成功率与预览稳定性。
2. 增加记录状态机，让列表明确显示处理中、可播放、失败。
3. 增加可配置采集参数，如 FPS、曝光和回溯时长。
4. 评估是否需要进一步从 raw frame ring buffer 升级到压缩后缓存方案。
